Cross-site scripting (XSS) is a vulnerability found in web applications that allows code (typically JavaScript) to be injected by attackers into web pages viewed by other users. The impact of XSS depends on the circumstances surrounding the vulnerability but can include cookie theft and page manipulation. In the former case, the attacker can masquerade as the victim user by making use of stolen session cookies; in the latter case, the attacker can perform a phishing attack, enticing the victim user to enter confidential information. The following scenario breaks down how a XSS attack could work:

  1. Alice has an account with a website hosted by Bob.
  2. Eve observes that Bob's website contains a reflected XSS vulnerability.
  3. Eve crafts a URL to exploit the vulnerability and sends an email to Alice amongst many others. The email, spoofed to look like it originates from Bob, uses social engineering to persuade Alice to click on the link.
  4. Alice clicks the link while logged in to Bob's website. (If Alice wasn't logged in at the time, Bob's website would likely redirect Alice to a login page and, following successful authentication, Alice would likely be returned to the vulnerable URL.)
  5. Eve's malicious script embedded in the link is returned in the response from Bob's website and executes in Alice's browser, as if it had originated from Bob. The script sends Alice's session cookie to Eve, who uses it to access Alice's account. Eve can view Alice's data and perform any action for which Alice has authorisation.

The above example illustrates non-persistent (or reflected) cross-site scripting when the malicious script within the request is immediately reflected back in the response. Note that only users clicking malicious links will be affected. More information and examples of XSS may be found at https://www.owasp.org/index.php/XSS.

XSS found within the Paragon application

The i2 application (http://vpadti2p.usfdc.corpintra.net:33001/i2/) is vulnerable to reflective XSS attacks. Following are details of the vulnerable pages and parameters that allow XSS to occur. It is possible that other pages are also vulnerable and any page that returns data originating from end-users should be reviewed in the light of the recommendations given in the main body of the report.
